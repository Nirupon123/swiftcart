# ADR 001: Stateless Authentication using JWTs and Redis Denylist

**Date:** March 27, 2026 

## Context & The Issue
SwiftCart is designed to handle massive concurrent traffic (targeting 1M DAU) and scale horizontally across multiple instances. 
Django's default authentication system uses server-side sessions stored in a relational database. Under heavy load, this creates a severe bottleneck: every single HTTP request requires a database hit just to verify the user's identity before processing the actual business logic (e.g., adding an item to a cart). 

## Decision
We will completely disable Django's default session middleware. 
Instead, we will implement **Stateless Authentication** using short-lived JSON Web Tokens (JWTs). 
* The user's identity and basic routing context (like `warehouse_region`) will be encoded cryptographically inside the JWT payload.
* To solve the "JWT Revocation Problem" (e.g., a user logging out before the token expires), we will maintain a **Token Denylist in Redis**. 

## Consequences

### Positive:
* **True Horizontal Scaling:** Django instances no longer rely on shared database state to authenticate users.
* **Massive Latency Reduction:** Database I/O is eliminated during authentication.
* **O(1) Revocation:** Checking the Redis denylist is an in-memory operation, taking fractions of a millisecond.

### Negative (Trade-offs to manage):
* **Increased Complexity:** We have to manually handle token expiration, refresh tokens, and Redis connections.
* **Infrastructure Dependency:** Authentication is now strictly dependent on Redis being highly available. If Redis goes down, we cannot verify if a token was revoked.
